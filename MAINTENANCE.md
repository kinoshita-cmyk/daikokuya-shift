# 保守・運用ガイド（大黒屋シフト管理システム）

最終更新: 2026-07-19

このドキュメントは「毎月のルール調整のたびにシステム改修とトラブルが発生する」
状態から脱するために導入した仕組みと、その使い方をまとめたものです。

---

## 1. 全体の仕組み（なぜ壊れにくくなったか）

### 1-1. ルール判定の一元化

月末月初の固定配置（大宮駅前固定・店長自店舗固定）の判定は、
**`prototype/rules.py` の `month_edge_forced_assignments()` に一元化**されています。

- 生成エンジン（`prototype/generator.py`）
- 検証エンジン（`prototype/validator.py`）
- 未完成下書き作成（`app/app.py`）

の3つは必ずこの共有関数を呼びます。以前は3か所に同じロジックが
別々に書かれており、1か所だけ直して矛盾する事故が起きていました。

共有関数の免除条件（この順で判定）:
1. 本人の×休み希望がある日 → 固定しない
2. 店舗が休店の日（例: 東口の月曜） → 固定しない
3. 前月からの連勤持ち越しで連勤上限を超える月初1日 → 固定しない
   （月例外設定で境界連勤の延長が許可されている人は固定を維持）

条件3により、**「前月末に5連勤で終えた人が月初固定と衝突して解なし」
という2026年8月型の障害は、今後は自動で回避**されます（免除した場合は
生成画面にその旨が表示されます）。

### 1-2. 月限定の例外はコードでなく設定ファイルへ

月限定の例外は **`config/monthly_exceptions.json`** に書きます。
コードの修正・アップロードは不要です（JSONのアップロードは必要）。

| キー | 意味 |
|---|---|
| `omiya_anchor_relaxed_months` | 大宮アンカー条件を緩和する月 |
| `carryover_consecutive_allowances` | 前月境界の連勤上限を人別に延長する月 |
| `avoid_same_off` | 指定2名の同時休みを避ける月限定ソフトルール |

例: 「2026年9月も下地さんの境界連勤を1日延長したい」場合は
`"carryover_consecutive_allowances"` に `"2026-09": {"下地": 1}` を追記して
GitHub にアップロードするだけです。

ファイルが壊れている場合はコード内デフォルトで動き続けます。
読み込み状態は経営者画面の「🔧 診断」に表示されます。

## 2. トラブル時の調べ方（解なし・生成失敗）

### 2-1. 解なしの原因自動調査

シフト生成が「未完成下書き」になった場合、画面に
**「🔬 解なしの原因調査」** が自動で表示されます。

1. **即時検査**（自動実行済み）:
   「○○さんの9月1日: 出勤希望と持ち越し休みが衝突」のように、
   既知パターンの矛盾を具体的な人・日付つきで表示します。
2. **条件別の自動調査**（ボタンで実行、2〜3分）:
   ×休み・出勤希望・固定配置・持ち越しなどの条件を1つずつ外して
   再計算し、「どの条件を外すと解が出るか」を特定します。

これまで人間とAIが手作業でやっていた総当たり調査の自動化です。
**解なしが起きたら、まずこの2つの結果を見てから対処を決めてください。**

### 2-2. 将来月の事前チェック

経営者画面の **「📅 将来月の事前チェック」** で、今後6〜12ヶ月分の
「基本ルールだけで生成可能か」を検査できます。

- 月末・月初が月曜（東口休店）に当たる月のルール衝突などを
  **その月が来る前に**発見できます
- 月替わりのタイミングで1回実行するのがおすすめです
- ローカルでも実行できます: `python3 -m prototype.preflight_calendar_check`

## 3. ファイル変更時の手順（重要）

過去に「ローカルで修正したのに GitHub へのアップロードが漏れて、
直したはずの障害が再発する」事故が複数回起きています。
**ファイルを変更したら必ず以下の手順で反映してください。**

```
1. ローカルでファイルを修正する
2. プロジェクト直下で同期チェックを実行する:
       python3 check_github_sync.py
3. 「❌ 差分あり」「⚠ GitHubに無い」と表示された全ファイルを
   GitHub にアップロードする（フォルダの対応に注意）
4. 1〜2分待って、もう一度同期チェックを実行する:
       python3 check_github_sync.py
   → 「🎉 全ファイルが GitHub と一致」が出るまで繰り返す
5. Streamlit Cloud をリブートし、アプリの動作を確認する
```

- アップロードは**変更したファイルだけでなく、チェックで検出された
  全ファイル**を対象にしてください（自分以外の修正が残っていることがあります）
- 複数のAI（Codex / Claude 等）に修正を頼んでいる場合は、
  作業の切り替わりのたびに同期チェックを実行すると安全です

## 4. 定期メンテナンスのチェックリスト

### 毎月（シフト作成前）
- [ ] 「📅 将来月の事前チェック」を実行し、来月〜半年先に ❌ がないか確認
- [ ] 提出締切後、「シフト自動生成」を実行
- [ ] 解なしになったら「🔬 解なしの原因調査」の結果を確認して対処

### ファイル変更のたび
- [ ] `python3 check_github_sync.py` で差分ゼロを確認

### 月限定の特例が発生したら
- [ ] コード修正ではなく `config/monthly_exceptions.json` への追記で対応
- [ ] 追記後、JSONファイルを GitHub にアップロード
